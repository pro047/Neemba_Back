### 설계 결정과 한계 (다음 세션이 알아야 할 것)

- **UptimeRobot 대신 자체 구현을 택한 이유**: 코드 소유 + 외부 SaaS 의존 제거. 대가는 **감지 지연** — GHA cron은 5분 설정해도 실제 5~30분 지연 흔함. "예배 중 즉시 감지"가 요구되면 UptimeRobot 병행 재검토.
- **감시는 반드시 서버 밖에서**: 서버 안 감시는 서버와 함께 죽음(cert-watch와 동일 원칙). 맥 로컬 데몬안은 "맥 꺼짐 = 감시 공백을 스스로 못 알림"(dead man's switch 필요)이라 기각.
- 실패 시 알림 경로 2중: GitHub 실패 메일(하한선, 시크릿 무관) + 디스코드 웹훅(시크릿 있을 때). 시크릿 미설정이면 warning 후 스킵이라 웹훅이 조용히 죽어도 메일은 옴.
- **중복 억제 없음**: 서버가 오래 죽어 있으면 5분마다 디스코드 알림 반복. 시끄러우면 그때 dedup 추가.
- 60일간 리포 활동 없으면 GHA 스케줄 자동 정지(현 커밋 빈도상 무관하나 장기 방치 시 유의).
- **웹훅 URL이 채팅에 1회 노출됨**(로컬 세션이라 저위험). 유출 의심 시 디스코드에서 재발급 → 시크릿만 갱신.
