### 4-2. RTMP 시작 10~20초 지연 후 burst 발행 (2026-07-19 진단, 미적용 — §4-3·4-4 다음 순위)

**증상**: 앱에서 스타트 후 10~20초간 무반응 → 모였던 메시지가 node에서 한번에 publish → python도 한번에 consume. 이후엔 정상. python은 원인 아님(`fetch(batch=5)`는 쌓인 만큼 즉시 가져오는 거울일 뿐).

**원인 가설 (코드 분석 완료, 서버 검증만 남음)**: `services/node/src/adapters/FfmpegTranscoder.ts`의 `-analyzeduration 0`이 함정값 — ffmpeg에서 0은 "분석 생략"이 아니라 "기본값 사용"이고 RTMP(FLV) 입력의 기본 스트림 분석 시간은 **90초**. 실제 프로브 종료 조건이 `-probesize 32k`(32KB)가 되는데, 저비트레이트 음성(AAC 모노 16~32kbps ≈ 2~4KB/s)이면 32KB 차는 데 8~16초. 프로브 동안 ffmpeg는 출력 없이 내부 버퍼링만 하다가(`-fflags nobuffer`는 이 단계 무효) 종료 순간 한꺼번에 방류 → STT에 실시간보다 빠르게 유입 → 결과 burst → publish/consume burst. 32,768B ÷ 10~20s ≈ 13~26kbps로 증상과 계산 일치.

**검증 (서버에서, 수정 전 필수)**:
```bash
# ① 재현 관찰: ffmpeg 시작 ~ 첫 out_time 진행 간격
docker logs -f --timestamps node | grep -E "ffmpeg|configured|published"
# ② 격리: 현행 플래그로 첫 출력까지 시간
docker exec node sh -c 'time ffmpeg -hide_banner -analyzeduration 0 -probesize 32k -fflags nobuffer -i rtmp://rtmp:1935/live/translation -vn -t 3 -f null - 2>&1 | tail -5'
# ③ 대조군: analyzeduration 0.5초 — 지연 사라지면 확정
docker exec node sh -c 'time ffmpeg -hide_banner -analyzeduration 500000 -probesize 32k -fflags nobuffer -i rtmp://rtmp:1935/live/translation -vn -t 3 -f null - 2>&1 | tail -5'
# ④ 앱 발행 스트림 실체(코덱·비트레이트·비디오 트랙 유무)
docker exec node sh -c 'ffprobe -v error -show_streams rtmp://rtmp:1935/live/translation 2>&1 | head -30'
```
**반증 조건**: ②·③ 차이 없음, 또는 ④에서 비트레이트가 높아 32KB가 1초 내 충족 → 가설 기각. 차순위: 앱 RTMP 인코더 송출 버퍼링 / gRPC 연결 전 오디오 클라이언트 버퍼 적체.

**수정안 (확정 시 1줄)**: `FfmpegTranscoder.ts`의 `"-analyzeduration", "0"` → `"-analyzeduration", "500000"`. `-probesize 32k`는 상한이라 유지(과하게 줄이면 코덱 식별 실패 위험). TDD 절차대로 red→green 후 배포.

### 4-3. /ws 끊김 인지 지연으로 최대 30초 번역 유실 (2026-07-19 예배 중 관찰, 미수정)

**증상 로그**: `main : websocket disconnected` 직후 `hub: send failed: Unexpected ASGI message 'websocket.send', after ...` ×6 → 약 30초 뒤 keepalive가 감지하고서야 `hub: queued send ... pending=N`으로 전환. 그 사이 문장 6개 유실.

**원인 (코드 확정, 3중)**:
1. `main.py` /ws 엔드포인트의 `except WebSocketDisconnect:`가 `pass`만 함 — hub에 통지 없음 → hub는 다음 keepalive 틱(최대 30s)까지 죽은 소켓을 슬롯에 유지.
2. starlette는 클라 주도 끊김 시 `client_state`만 DISCONNECTED로 바꾸고 `application_state`는 CONNECTED로 남음 → `websocket.py`의 `broadcast_to_session`/`_send_text` 실패 핸들러의 `application_state != CONNECTED` 검사가 통과/무효 → 슬롯 미정리 상태로 send 반복 시도.
3. `_send_text` 실패 시 해당 문장을 pending에 재적재하지 않음 → 큐잉 없이 즉시 유실.

**끊김 발원지는 python 아님**: 서버발 close 경로(keepalive 타임아웃)는 "no pong for Ns" 로그를 반드시 남기는데 부재. nginx도 배제(`/ws`는 read 3600s + 30s ping/pong 트래픽). 앱 백그라운드 전환·망 전환 추정. 재접속 후 `_flush_pending` 방류는 정상 동작 확인.

**수정안 (TDD)**: ① disconnect 핸들러에서 hub에 즉시 통지 — 슬롯 비움 + reconnect 대기 전환. **`detach`는 pending을 비우므로 쓰지 말 것** — `_mark_waiting_for_reconnect` 성격의 공개 메서드 신설. ② 연결 검사에 `client_state` 병용. ③ send 실패 문장 pending 재적재.

### 4-4. 2026-07-19 예배 장애 후속 — **§4-3과 함께 다음 세션 최우선** (전말: `docs/incidents/2026-07-19-network-blip-ws-rtmp.md`)
폰 네트워크 순단 1회 → ws·RTMP 동시 절단 → 복구 로직 부재로 장애 지속. 수정 3건:
1. **앱 RTMP 송출 재연결** (몸통): 끊김 감지 → 백오프 재송출. ws `connectWithRetry`와 동일 패턴. "rtmp mounted 가드" WIP 정리와 함께. 성공 판정 = node `rtmp on_publish` 로그.
2. **node STT 좀비화 완화** (`StreamOrchestrator`): "입력 없음"을 STT 에러로 카운트해 4연속 시 영구 포기 → ffmpeg 무한 재시도와 어긋남. 오디오 재유입 시 STT 재개되게 (예: giving-up 대신 로테이션 일시정지 + 오디오 감지 시 재개).
3. **앱 ws 재시도 예산 증대** (`ws_client.dart`): `maxRetries 2`(~3초 포기) → 서버 5분 대기·pending 큐와 균형 맞춰 확대 (백오프 상한 포함).

**전송 방식 결정 (2026-07-19 사용자 확정)**: WebSocket 유지. 폴링 대안은 +0.5~1초 지연 때문에 기각, SSE는 자리가 애매해 기각. 그리고 **다중 청중(여러 폰 동시 시청) 확장 예정** — 현 `WebSocketHub`는 동시 1클라이언트 슬롯 전제이므로, §4-3 수정 시 **클라이언트별 상태(연결·pending 큐)를 per-connection 구조로 설계**해 단일 슬롯에 갇히지 않게 할 것. 다중화 본작업은 /ws 인증(§4-6)과 얽히므로 별도 세션에서.
