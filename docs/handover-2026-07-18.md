### 4-2. RTMP 시작 10~20초 지연 후 burst 발행 (2026-07-19 로컬 검증으로 원인 확정·수정 — 배포 대기)

**증상**: 앱에서 스타트 후 10~20초간 무반응 → 모였던 메시지가 node에서 한번에 publish → python도 한번에 consume. 이후엔 정상. python은 원인 아님.

**로컬 검증 결과 (2026-07-19, dev 스택 + ARM 네이티브 nginx-rtmp rig)** — 기존 analyzeduration 단독 가설은 **부분 기각**, 진짜 몸통은 따로 있었다:

1. **몸통 = nginx-rtmp 워커 복권**: `infra/rtmp/nginx.conf`가 `worker_processes auto`인데 `rtmp_auto_push` 없음. nginx-rtmp는 스트림을 publisher가 붙은 워커 안에만 유지하므로, 다른 워커에 붙은 구독자(node ffmpeg pull)는 **영원히 0바이트** 수신. 로컬 실측: 멀티 워커에서는 기본 플래그 ffprobe조차 무한 대기, `worker_processes 1`로 바꾸면 즉시·반복 성공(3/3). prod에선 pull이 잘못된 워커에 걸리면 10s 스톨 → 재시작 → 맞는 워커에 걸릴 때까지 반복 = 10~20초 지연 + "가끔은 즉시 되는" 무작위성까지 설명. → **수정: `worker_processes 1`** (rtmp_auto_push 는 로컬 검증에서 소켓만 생기고 중계 안 됨 — 신뢰 불가로 배제. 단일 스트림 워크로드라 워커 1개면 충분).
2. **부차 = analyzeduration 0 함정**: 0은 "분석 생략"이 아니라 "포맷 기본값"이라 프로브가 probesize 종료 조건에 매달림. 정상 rig 실측(24kbps 오디오): 현행 ~2s vs 500000 ~0s. 원래 가설의 10~20초 규모는 아니지만 재접속마다 반복되는 실지연이므로 **수정: `"-analyzeduration", "500000"`** (`-probesize 32k`는 상한이라 유지).
3. **보너스 = 재송출 후 백오프 공백 제거**: OBS가 순단 후 재송출해도 ffmpeg는 백오프 타이머(10s→…→60s)가 돌아올 때까지 대기했다. nginx-rtmp `on_publish` 훅이 "publisher 복귀" 신호 그 자체이므로, 훅 허용 시 대기 중인 재시작을 즉시 발화(`FfmpegTranscoder.restartNow`, rtmp.ts 배선). 세션 없거나 정상 동작 중이면 no-op.

**남은 확인 (배포 후 prod에서)**: 첫 스타트~첫 번역까지 체감 지연이 수 초 내로 줄었는지, `ffmpeg process not updated for 10s` 반복이 사라졌는지. OBS 실스트림(비디오 포함)에서 ④ `docker exec node sh -c 'ffprobe -v error -show_streams rtmp://rtmp:1935/live/translation'` 로 스트림 실체도 한번 기록해둘 것.

### 4-3. /ws 끊김 인지 지연으로 최대 30초 번역 유실 (2026-07-19 예배 중 관찰, 미수정)

**증상 로그**: `main : websocket disconnected` 직후 `hub: send failed: Unexpected ASGI message 'websocket.send', after ...` ×6 → 약 30초 뒤 keepalive가 감지하고서야 `hub: queued send ... pending=N`으로 전환. 그 사이 문장 6개 유실.

**원인 (코드 확정, 3중)**:
1. `main.py` /ws 엔드포인트의 `except WebSocketDisconnect:`가 `pass`만 함 — hub에 통지 없음 → hub는 다음 keepalive 틱(최대 30s)까지 죽은 소켓을 슬롯에 유지.
2. starlette는 클라 주도 끊김 시 `client_state`만 DISCONNECTED로 바꾸고 `application_state`는 CONNECTED로 남음 → `websocket.py`의 `broadcast_to_session`/`_send_text` 실패 핸들러의 `application_state != CONNECTED` 검사가 통과/무효 → 슬롯 미정리 상태로 send 반복 시도.
3. `_send_text` 실패 시 해당 문장을 pending에 재적재하지 않음 → 큐잉 없이 즉시 유실.

**끊김 발원지는 python 아님**: 서버발 close 경로(keepalive 타임아웃)는 "no pong for Ns" 로그를 반드시 남기는데 부재. nginx도 배제(`/ws`는 read 3600s + 30s ping/pong 트래픽). 앱 백그라운드 전환·망 전환 추정. 재접속 후 `_flush_pending` 방류는 정상 동작 확인.

**수정안 (TDD)**: ① disconnect 핸들러에서 hub에 즉시 통지 — 슬롯 비움 + reconnect 대기 전환. **`detach`는 pending을 비우므로 쓰지 말 것** — `_mark_waiting_for_reconnect` 성격의 공개 메서드 신설. ② 연결 검사에 `client_state` 병용. ③ send 실패 문장 pending 재적재.
