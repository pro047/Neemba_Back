## 4. 다음 작업 (우선순위 순)

### 4-1. 다음 예배 실전 검증 — **최우선**
- 배포된 수정의 prod 확인: 세션 시작~첫 번역 **수 초 내** / 끊김 시 `hub: client disconnected` 즉시 로그 + 재접속 시 pending 방류 / `Stt paused`→오디오 복귀 시 `Stt resuming` / `ffmpeg process not updated for 10s` 반복 소멸.
- OBS 실스트림 실체 1회 기록: `docker exec node sh -c 'ffprobe -v error -show_streams rtmp://rtmp:1935/live/translation'`.
- 순단 시 운영 절차: obs-multi-rtmp 스트림 토글 → 안 되면 OBS 완전 재시작. 복구 판정 = node 로그 `rtmp on_publish`.

### 4-2. RTMP 인증 활성화 — **사용자 작업, 우선순위 상향** (현재 아무나 publish 가능)
1. OBS 스트림 키: `translation` → `translation?key=<비밀값>`
2. 로컬 `.env.prod`에 `RTMP_PUBLISH_KEY=<비밀값>` 추가 → `gh secret set ENV_PROD -R pro047/Neemba_Back < .env.prod`
3. 다음 배포부터 강제. 켠 뒤 node 로그에 `auth disabled` 경고 보이면 env 누락 — 즉시 조치.

### 4-3. mvp 정리
사용자 WIP(feature/mic) 커밋 — ws 재시도 12회 수정 동반. 이후 mvp main-develop 구조 세팅.

### 4-4. backpressure 부재 (구 §4-5)
NATS 중단·소비 지연 시 span 드롭. 착수 시 **선택지 표부터**(버퍼링·재시도·유실 허용 트레이드오프) 만들어 사용자 결정 후 구현. consumer ack 시점 이동과 함께 검토.

### 4-5. /ws 인증 + 다중 청중 허브 재설계 (구 §4-6, 얽힘)
현 `WebSocketHub`는 단일 슬롯(동시 1클라) — 다중 청중(여러 폰 동시 시청) 로드맵에 따라 **클라이언트별 상태(연결·pending)의 per-connection 구조**로 재설계. 앱(Flutter) 수정 필요라 mvp WIP 정리 후 한 세션에서.

### 4-6. IAM 최소권한 분리 (구 §4-7, **상향 권고**)
`ljs`(AdministratorAccess) 단일 키가 로컬 에이전트 + GHA 시크릿 양쪽에서 사용 중 — 유출 시 계정 전체 노출. 읽기 전용 + 필요 시 상승 구조로 분리, 테라폼으로.

### 4-7. 백로그 (구 §4-8·4-9 통합)
